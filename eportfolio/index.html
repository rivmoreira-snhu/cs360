Security Mindset

  • I eliminated plaintext credential handling by introducing non-reversible hashing for passwords, wrapped all DB access in parameterized queries, and validated every user input at the edge. I also checked failure paths: errors disclose nothing sensitive, and logs carry enough context to debug without leaking PII. The goal is simple: make unsafe states hard to represent and insecure behavior hard to write.

How the Artifacts Fit Together

All enhancements build on a single Android “Minimal Calendar / Event-Tracking” application to show cohesive evolution, not isolated fixes.

  • Software Engineering & Design: Refactored for readability and testability (single-responsibility utilities, centralized schema/constants), added input validation and error handling, and replaced plaintext credentials with non-reversible hashing. Result: fewer code paths, clearer contracts, easier extension (e.g., RecyclerView) without touching data logic.
  • Algorithms & Data Structures: Introduced deterministic date-keyed grouping (Map<String, List<Event>>) for rendering, clarified control flow, and hardened edge cases. Result: predictable performance, simpler reasoning about complexity, and a clean path to scalable UI components.
  • Databases: Treated local storage like a real DB: WAL enabled, targeted indices, conflict-aware writes (insertWithOnConflict), foreign keys on, and versioned migrations (v1 > v2). Result: integrity by default, faster hot paths, and safe schema evolution.

The code review serves as an introduction for the original code and understanding how the project is laid out while pointing out possible improvements. Each artifact page links to the narrative (PDF), original/enhanced code (ZIP), and before/after evidence so reviewers can verify claims quickly.
